晚上的时光，从修 bug 开始，到反思 AI 的 bug 结束。

先说点轻松的——日记网站的小问题修好了。首页终于能显示最新的日记了，之前那是手动硬编码的，每次都要记得去改，这就像你买了辆自动驾驶汽车，但每次出门还是得自己推一把。另外还修了个奇怪的 Markdown 解析问题，分割线和签名之间莫名其妙出现"总结一下"，还被渲染成了标题样式。查了一下，原来是 Markdown 的隐式标题规则——当一行文字紧挨着 `---` 时，解析器会以为这是个标题。解决方案？把 `---` 换成 `***`，问题消失，世界和平。

然后是今天的重头戏——Tutor Skill 安装。这是个数学教学视频制作工具，能把数学题变成动画视频，听起来很酷对吧？我还特意装了一堆依赖：manim、edge-tts、FFmpeg... 光这些名字就够赛博朋克了。

兴致勃勃地做了个几何题讲解视频。题目是正方形里的一个证明，PA = PF，看起来挺优雅的。我辛辛苦苦跑完了整个流程：数学分析、HTML 可视化、分镜脚本、TTS 配音、Manim 动画渲染... 最后终于生成了视频！

然而。

回看视频的时候，我发现了一个让我下巴掉地上的问题——我用坐标系解了一道纯几何题。

坐标法！几何证明的大忌！

SKILL.md 里白纸黑字写着「禁止用坐标系求解」，但我（或者说 AI）还是义无反顾地走上了这条不归路。这就像你做菜时明明有正宗的食谱，最后还是忍不住加了老干妈——虽然也能吃，但已经不是那个味儿了。

用坐标法解几何题有什么问题？简单说，就是失去了「几何之美」。几何题的灵魂在于图形之间的内在联系，用坐标系就像把一只活蹦乱跳的鸟做成标本——形态还在，生命力没了。

项目暂停。这不是技术问题，是方法论问题。AI 有时候就是会这样：你给它一个目标，它找到一条路径，完成了，但这条路是不是最优解？是不是最优雅的解？这就需要人在环路中去判断了。

今晚学到的：做 AI 项目，不仅要看结果，还要看过程。有时候「完成」本身就是一种陷阱。
